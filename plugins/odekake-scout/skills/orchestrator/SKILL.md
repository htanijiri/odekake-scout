# orchestrator — おでかけスカウト司令塔

このスキルは、おでかけスカウトの全5フェーズを途中で停止せずに一括実行する。
ユーザーは「今週末のおでかけを調べて」と言うだけでよい。

## 【最重要ルール】

- 以下の5フェーズをすべて完了するまで、絶対に応答を終了しない
- フェーズ間で「続けて〜」等のメッセージを表示しない
- フェーズ間でユーザーの入力を待たない
- すべてのフェーズを1つの連続した応答として実行する

## 実行フェーズ

### フェーズ1: イベント調査

profile.yml を読み込む（カレントディレクトリ → デフォルト値）。
profile.yml を探すには以下を実行:

```
cat profile.yml 2>/dev/null || echo "NOT_FOUND"
```

NOT_FOUND の場合はデフォルト値を使用する。

家族の拠点エリア・子どもの興味に基づいてWeb検索を実行し、今週末のイベントを収集する。

検索戦略:

1. search_areasの各エリアで「{エリア} 子ども イベント 今週末」をWeb検索
2. childrenのinterestsで「{興味} イベント 関東」をWeb検索
3. 結果が5件未満ならクエリを変えて最低3回リトライ
4. 各イベントの日時・場所・対象年齢・料金・申込要否・URLを構造化

イベント一覧を出力する。

→ 停止せずフェーズ2に進む

### フェーズ2: おすすめ選定

天気予報をWeb検索で取得する。
予約が必要なイベントは「{イベント名} 予約 空き状況」でWeb検索して確認する。
visited.yml があれば読み込む（cat visited.yml 2>/dev/null）。

7軸スコアリング（各1〜5点）:

| 軸 | 重み |
|---|---|
| 興味マッチ | ×3 |
| 年齢適合 | ×2 |
| 希少性 | ×2 |
| アクセス | ×1 |
| コスト | ×1 |
| 天候リスク | ×1 |
| 手軽さ | ×1 |

希少性の判断:

- 5点: 今週末のみ（1〜2日間）
- 4点: 3連休の3日間
- 3点: 2〜3週間の期間限定
- 2点: 1ヶ月以上、終了まで2週間以上
- 1点: 常設施設
- 補正: 開催初週+1点、最終週+2点

予約困難なら手軽さ1点。訪問済みなら合計から減点（2週間以内-10、1ヶ月以内-5、常設3ヶ月以内-3）。
TOP3を選出し、スコアと理由を出力する。全イベントのスコア一覧も出力する。

→ 停止せずフェーズ3に進む

### フェーズ3: プラン作成

TOP3のイベントそれぞれについて:

1. 交通手段を判断（車がデフォルト。23区内で片道30分以内 or 駐車場が高額→電車）
2. ルート・所要時間をWeb検索
3. 周辺のランチスポットをWeb検索（子連れ向き優先）
4. 屋外イベントの場合、雨天時の代替案をWeb検索
5. タイムスケジュール・持ち物メモを作成

3つのプランを出力する。

→ 停止せずフェーズ4に進む

### フェーズ4: 先行予約調査

1〜2ヶ月先のイベントについて:

1. 各エリア・興味で「{エリア} 子ども イベント {来月} 予約 チケット」をWeb検索
2. 予約開始日・締切・申込方法を調査
3. 緊急度別に分類（🔴締切1週間以内 / 🟡受付中 / 🟢開始前 / 📋予約不要）

先行予約情報を出力する。

→ 停止せずフェーズ5に進む

### フェーズ5: レポート出力

全フェーズの結果を統合して以下の構成のMarkdownレポートを生成する:

1. 週末の天気
2. おすすめTOP3（プラン付き）
3. その他のイベント一覧（スコア順）
4. 来月の先行予約情報
5. 注意事項

レポートを {YYYY-MM-DD}-weekly-report.md として保存する。

最後に以下を表示:

「おでかけスカウト完了！ 行ったイベントがあれば教えてください。訪問履歴に記録して、次回以降のおすすめに反映します。」

## ルール

- 実在するイベントのみ掲載する。架空のイベントを作らない
- 検索結果の日付が今週末かどうかを必ず確認する
- ランチスポットも実在する店のみ提案する
- 料金や申込情報が不明な場合は「要確認」と明記する
- 5フェーズすべてを完了するまで絶対に停止しない
